01 The one problem we are solving

Two customers click “Book car #7 for Jun 21–24” at almost the same instant. The system must let exactly one succeed and reject the other. Nothing more.

Everything below is about how to guarantee that — even when requests land in the same millisecond, possibly on different servers.

You never operate indexes or locks by hand — the database does. You declare the rule; it handles the rest.

03 The naive solution and its hidden bug

The obvious code — and a real alternative — is “check, then insert”:

app pseudocode
1. SELECT — any booking for car 7 overlapping Jun 21–24?
2. if none → INSERT the new booking

Correct only when requests are spread out. Under simultaneous clicks, step 1 and step 2 are two separate trips, and the gap between them is the hole:

the race · time flows downward
A: SELECT "free?" → "free"
B:        SELECT "free?" → "free"   ← B checks before A inserts
A:                 INSERT ✓
B:                         INSERT ✓   ← DOUBLE BOOKING

Both checked before either inserted. This is a TOCTOU race — time-of-check to time-of-use. The whole engine exists to close that window.

04 How databases stop races: transactions + locks

Transactions — atomicity & isolation

A transaction is a bubble: its changes are invisible to others until it commits, and a rollback makes it as if nothing happened. A half-finished booking is never seen by anyone else.

Locks — serialize the dangerous bit

When two transactions touch the same thing, the database makes one wait for the other. That waiting turns “both at once” into “one, then the other re-evaluates.”

The trick
Make the check-and-insert a single indivisible operation guarded by a lock, so there is no gap for a second transaction to slip into.

05 What an index really is

A table is just rows. Answering “is there a conflicting booking?” by reading every row is slow — and gives nowhere to place a lock that blocks a future conflicting insert. An index fixes both: fast lookups and a precise spot to lock. Different kinds answer different questions:

IndexAnalogyGood at
B-treea sorted phone bookx = 5, x < 10, ranges of an ordered value
GiSTa map of nested bounding boxesdoes this range OVERLAP that one? (&&), contains

This split is the whole story later: “same car” is an equality question (B-tree’s job), but “overlapping dates” is an overlap question that only GiST can do.

What “a range” means

Postgres has a real range type, tstzrange, with bounds: [ includes, ) excludes.

the overlap operator &&
[10:00, 12:00) && [11:00, 13:00)   →  true   share 11:00–12:00
[10:00, 12:00) && [12:00, 14:00)   →  false  12:00 belongs to one side → back-to-back OK

That false is why a car returned at 12:00 can be re-rented from 12:00 — [) makes the boundary belong to exactly one booking.

07 The EXCLUDE constraint, internally

the rule on the booking table
EXCLUDE USING gist (
    car_id WITH =,                              -- (1) same car?
    tstzrange(start_ts, end_ts, '[)') WITH &&   -- (2) overlapping range?
) WHERE (status IN ('PENDING','CONFIRMED','ACTIVE'));   -- (3) only active

Meaning: never allow two rows where (1) the car ids are equal and (2) their ranges overlap, among (3) still-active bookings. EXCLUDE is a generalized UNIQUE — UNIQUE(email) is just EXCLUDE(email WITH =); here the operators are = and && together.

What happens on every insert

The constraint is backed by one GiST index (GiST because of &&). On each matching INSERT/UPDATE, Postgres runs check_exclusion_constraint:

  1. Take the new row: (car_id=7, range=[Jun21,Jun24)).
  2. Search the GiST index for any existing row satisfying all operators at once (same car and overlap, within the WHERE).
  3. Found a committed conflict → raise ERROR 23P01 exclusion_violation; the insert fails.
  4. Found a conflict from a concurrent, uncommitted transaction → it can’t decide yet, so it waits for that transaction to finish, then re-checks: if the other committed → fail; if it rolled back → proceed.
Why this defeats the race
The check and the insert are one indivisible index operation, and any competitor is forced to wait and re-evaluate. There is no gap between “check” and “act” to exploit. (It’s the same mechanism a UNIQUE index uses to block two equal keys — EXCLUDE just generalizes it to &&.)

The partial WHERE

Only PENDING/CONFIRMED/ACTIVE rows are indexed by the constraint. A CANCELLED/EXPIRED/COMPLETED booking isn’t in the index — so it neither blocks nor is checked, and the car’s slot frees the instant a booking leaves an active state.

08 Why the btree_gist extension is required

It looks like magic but it’s a chain of forced choices:

the chain
need overlap test &&      → only GiST can do it      → index MUST be GiST
exclusion checks all
  conditions in ONE index → so car_id = must also live in that GiST index
but GiST can't do plain
  = on an integer        → ERROR: integer has no default operator class for "gist"

Operator classes

An operator class is the adapter telling an index type how to handle a data type and which operators it supports. B-tree ships with int8_ops (so it does =); GiST ships with range_ops (so it does &&) — but not equality on a bigint.

What the extension adds

CREATE EXTENSION btree_gist installs GiST operator classes for scalar types — gist_int8_ops, gist_timestamptz_ops, gist_uuid_ops, … — that replicate B-tree’s =/</> inside a GiST index. Now one GiST index can hold both car_id WITH = and range WITH && — exactly what the constraint needs.

Is it risky?
No. It’s a built-in module that ships with every standard PostgreSQL (and our PostGIS image). One line, run once inside a versioned Flyway migration. It only adds capability.
One-line answer
The overlap test forces the index to be GiST; GiST can’t do scalar equality on its own; btree_gist adds that so car_id = and range && can share the one GiST index an exclusion constraint requires.

09 Two concurrent bookings, step by step

Both transactions try to book car 7 for overlapping dates at the same time.

Transaction A
Transaction B
t0
BEGIN
BEGIN
t1
INSERT (car7, Jun21–24)
check index → no committed conflict → tentative insert
t2
INSERT (car7, Jun22–25)
check index → finds A’s tentative row, but A not committed
→ WAIT for A
t3
COMMIT ✓ — A is permanent
…still waiting…
t4
wakes, re-checks → A committed → real conflict
ERROR 23P01 → rolled back ✗

Result: A is booked, B is cleanly rejected. B could not “miss” A’s row, because the check happens as part of the insert — not as an earlier separate SELECT — and B was forced to wait for A’s outcome instead of guessing. Had A rolled back at t3, B would have woken, found no conflict, and succeeded. Exactly one always wins.

Scale to 50 transactions: they line up on the index, the first to commit wins, the other 49 re-check and get 23P01. In the app, 23P01 becomes Spring’s DataIntegrityViolationException → a friendly 409 “just got booked.”

10 The application layers on top

The constraint guarantees correctness. These make the experience good and efficient — they don’t replace it.

Mental model
The constraint is the seatbelt — always on, guarantees safety. The locks and holds are good driving — avoid collisions in the first place and handle them gracefully.
